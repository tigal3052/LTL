## 2. 비가역 규칙

### SoT
- 규칙 계산은 반드시 한 곳에만 둔다.
- 브라우저 경로와 headless 경로가 함께 있으면, 둘 중 하나를 SoT로 정하고 다른 쪽은 재사용만 한다.
- UI, HUD, 디버그 표시, 로그, 테스트 harness는 규칙을 다시 계산하면 안 된다.
- JSON 스키마, phase 흐름, 보상 규칙, 난이도 규칙도 각각 SoT를 명시한다.
- 규칙 설명과 주석도 SoT를 따라야 한다.

### LTL SoT 우선순위
- 핵심 규칙: `domain/` 또는 순수 규칙 모듈
- 진행 흐름: `process/`, `phases/`
- 데이터 구조: `data/*.json` + validator
- 밸런스 수치: tuning 문서 또는 데이터 파일
- UI: 표현 전용. SoT가 될 수 없다.

### 논리기반 프로그래밍
- 기능은 먼저 플레이어 관점 문장으로 쓴다.
- 문장을 phase 단위로 나눈다.
- 각 phase를 동사 함수와 상태 객체로 분해한다.
- 권장 순서: `process -> phases -> vocabulary -> domain/models -> ui`
- UI는 입력 전달, 상태 표시, 디버그 토글만 담당한다.

### 주석 규칙
- 주석은 코드보다 먼저 작성하며, 구현은 그 주석 구조를 따라야 한다.
- 주석은 현재 코드가 수행하는 역할을 한글 한 문장으로 설명해야 한다.
- 주석은 추상 요약이 아니라 실제로 다루는 대상, 상태, 값, 결과를 포함해야 한다.
- 주석 단위는 함수, 상태 초기화 묶음, phase 내부 주요 단계 묶음이다.
- 하나의 phase 함수 안에 여러 논리 단위가 있으면 각 논리 단위마다 별도 한글 주석이 있어야 한다.
- 주석은 두 종류만 허용한다.
- `직접 구현 주석`: 바로 아래 코드가 그 규칙을 직접 수행한다.
- `호출 중심 주석`: 바로 아래 코드가 특정 SoT 함수나 phase helper를 호출해 그 규칙을 위임한다고 설명한다.
- 한 주석이 직접 구현되지 않으면 반드시 호출 중심 주석으로 바꾼다.
- 세부 규칙 설명은 실제 규칙 SoT 함수 안에만 둔다.
- 상위 phase와 하위 helper가 같은 세부 규칙을 중복 설명하면 안 된다.
- 같은 세부 규칙이 여러 phase에서 필요하면 별도 함수 하나를 만들고 그 함수만 참조한다.
- 주석은 append 하지 말고 현재 최종 구현 기준으로 갱신한다.

## 3. 구현 규칙

- 프로토타입 검증이 끝난 규칙은 다시 추측하지 않는다.
- P단계에서는 화면보다 재현 가능한 headless 검증을 우선하되, 규칙 SoT는 하나만 유지한다.
- 랜덤은 반드시 seed 주입 방식으로 관리한다.
- 핵심 규칙은 순수 함수 또는 독립 객체로 두고, `Node` 의존은 컨트롤러/UI 계층으로 제한한다.
- 디버그 기능은 본편 규칙과 분리하고, 디버그 ON/OFF가 가능해야 한다.
- 미구현 항목은 TODO로 남기지 말고 실행 계획 또는 기술 부채 문서로 이동한다.

## 4. Ready

- 이번 작업의 SoT가 무엇인지 한 줄로 말할 수 있다.
- 기능을 플레이어 관점 문장으로 설명할 수 있다.
- 그 문장을 phase 단위로 나눌 수 있다.
- 직접 구현 주석과 호출 중심 주석을 구분해 phase 흐름을 먼저 적을 수 있다.
- fixture, replay, seed, 기대 결과 중 최소 하나의 검증 기준이 있다.
- 필요한 환경과 실행 방법이 `docs/15_pc-environment/`에 정리되어 있다.

## 5. Done

- 같은 규칙이 두 경로에 중복 구현되어 있지 않다.
- phase, 데이터, 규칙, UI의 책임이 섞이지 않는다.
- 세부 규칙 설명은 SoT 함수 하나에만 존재한다.
- 상위 phase 주석과 하위 helper 주석이 같은 세부 규칙을 중복 설명하지 않는다.
- headless 또는 브라우저 검증 결과가 남아 있다.
- 활성 exec-plan이 갱신되었고 완료 시 `docs/11_exec-plans/02_completed/`로 이동한다.
- 남은 보류 항목은 기술 부채 문서에 이유와 후속 시점을 적는다.
