# AGENTS.md

이 문서는 모든 하위 harness가 공통으로 따르는 핵심 규칙과 빠른 참조용 문서 지도를 정의한다. 구현자는 아래 규칙을 직접 따라야 하며, 프로젝트별 harness는 이 문서를 삭제하거나 축소하지 말고 현재 프로젝트 구조에 맞게 최신화한다.

## 1. 비가역 규칙

### SoT
- 모든 핵심 규칙은 반드시 하나의 **단일 진실 소스(Single Source of Truth, SoT)** 에만 존재해야 한다.
- UI, 테스트, 디버그, 어댑터, 임시 스크립트는 SoT를 재구현하면 안 된다.
- 같은 규칙이 두 곳에 있으면 하나를 SoT로 지정하고, 나머지는 그것을 호출하거나 참조하도록 정리한다.
- SoT는 코드뿐 아니라 규칙 설명과 주석에도 적용한다.

### 논리기반 프로그래밍
- 모든 코드는 문장으로 설명할 수 있어야 한다.
- 기능은 먼저 하나의 문장으로 정의한다.
- 큰 함수는 허용되지만, 내부 기능은 각각 별도의 문장으로 설명 가능한 작은 함수로 분리한다.
- 각 함수는 최종적으로 “한 문장을 수행하는 단위”가 되도록 설계한다.
- 구현 전에는 먼저 주석으로 논리 단위를 적고, 그 주석 순서대로 코드를 작성한다.
- 권장 순서: `문장 -> 절/단계 -> 작은 문장 함수 -> 객체/상태 -> UI/입출력`
- UI는 문장을 실행하는 곳이 아니라, 문장 실행을 요청하고 결과를 표시하는 곳이다.

### 주석 규칙
- 주석 규칙 SoT는 `[docs/comment-implementation-rules.md](../docs/comment-implementation-rules.md)` 이다.
- 이 파일의 주석 규칙은 위 문서를 축약 참조하며, 충돌이 나면 항상 위 문서를 우선한다.
- 함수 본문 안의 실행 주석은 반드시 `주석A -> 주석A 구현 -> 주석B -> 주석B 구현` 순서만 허용한다.
- `주석A -> 주석B -> 주석A 구현 -> 주석B 구현` 형태는 금지한다.
- 실행 주석은 반드시 바로 아래의 연속된 코드 블록 하나와만 1:1 대응해야 한다.
- 두 개 이상의 실행 주석이 하나의 코드 블록을 공유하려 하면 블록을 쪼개거나 helper를 추출한다.
- 기존 주석 스캐폴드가 있는 파일은 전체 교체하지 말고 부분 수정으로 구현한다.
- 기존 주석 문장은 구현 편의 때문에 임의 삭제, 합치기, 요약하지 않는다.

### 계획 우선
- 구현 전에 범위, SoT, 테스트 방법, 완료 기준을 먼저 문서로 고정한다.
- 구현 중 새 결정이 생기면 코드보다 문서를 먼저 갱신한다.

## 2. 작업 순서

1. `docs/09_PROJECT_INTAKE.md` 확인
2. 로컬 환경 문서 확인: `docs/15_pc-environment/`
3. 기능 명세 확인: `docs/12_product-specs/`
4. 아키텍처/설계 문서 확인
5. 활성 exec-plan 작성 또는 갱신: `docs/11_exec-plans/01_active/`
6. 테스트 작성 또는 검증 기준 확정
7. 구현
8. 검증 후 active plan을 completed로 이동

## 3. 구현 기준

- 구조를 먼저 읽고 수정한다. 추측으로 새 구조를 만들지 않는다.
- 반복해서 틀리는 작업은 수동 반복 대신 문서, 스크립트, 테스트로 고정한다.
- 템플릿 값과 실제 값을 섞지 않는다.
- 환경 의존 경로, 계정, SDK, 시크릿은 반드시 환경 문서에 기록한다.
- 승인 없는 대규모 구조 변경, 외부 서비스 도입, 운영 데이터 변경은 금지한다.


## 5. Document Structure

범용 harness는 프로젝트별 harness로 복사되거나 참조되는 기본 문서 구조를 제공한다. 없는 문서는 무리하게 가짜 내용으로 채우지 말고, 현재 프로젝트에 실제로 필요한 범위부터 작성한다.

```text
agent-harness/
├── README.md                         # harness 사용법과 복사/적용 안내
├── 00_AGENTS.md                      # 모든 하위 harness가 따르는 공통 규칙과 문서 지도
├── 01_ARCHITECTURE.md                # 범용 harness의 기술 구조와 확장 원칙
└── docs/
    ├── 00_PRODUCT_SENSE.md           # 제품 감각, 사용자 가치, 판단 기준
    ├── 01_PLANS.md                   # 장기 계획과 작업 흐름 요약
    ├── 02_DESIGN.md                  # 설계 방향과 선택지
    ├── 07_TESTING.md                 # 테스트 전략과 검증 명령
    ├── 08_DEPLOYMENT.md              # 배포 기준과 릴리스 절차
    ├── 09_PROJECT_INTAKE.md          # 문제 정의, 범위, 성공 기준, Ready 확인
    ├── 09_OPERATIONS.md              # 운영, 모니터링, 장애 대응 기준
    ├── 11_exec-plans/
    │   ├── 00_tech-debt-tracker.md   # 기술 부채와 후속 결정 추적
    │   ├── 01_active/                # 현재 실행 중인 계획
    │   │   └── _TEMPLATE-active.md
    │   └── 02_completed/             # 검증 완료된 계획과 결과
    │       └── _TEMPLATE-completed.md
    ├── 12_product-specs/
    │   └── 00_index.md               # 기능 명세 색인
    ├── 14_references/
    │   └── 00_README.md              # 외부 참고자료 색인
    └── 15_pc-environment/
        └── 00_README.md              # 로컬 환경 문서 색인
```

## 6. Workflow Gates

### Intake Gate
- 문제 정의, 목표, 제외 범위, 승인자, 성공 기준이 없으면 구현하지 않는다.

### Planning Gate
- `docs/09_PROJECT_INTAKE.md`의 Ready Check를 충족한다.
- 로컬 툴체인 의존 작업이면 `docs/15_pc-environment/`에 실제 문서가 존재한다.
- 필요한 경우 `docs/12_product-specs/`에 기능 명세가 존재한다.
- `docs/11_exec-plans/01_active/`에 기능 단위 실행 계획을 생성한다.
- 계획에는 수정 대상 파일, 파일별 변경 내용, 검증 단계, 가정 및 미결정 항목이 있어야 한다.

### Implementation Gate
- 구현 전 주석으로 기능 문장과 논리 단위를 먼저 적는다.
- 직접 구현 주석과 호출 중심 주석을 구분한다.
- SoT가 아닌 계층은 규칙을 재구현하지 않고 SoT를 호출하거나 참조한다.
- 코드 변경 중 새 결정이 생기면 코드보다 문서를 먼저 갱신한다.

### Verification Gate
- 테스트 또는 사전에 확정한 검증 절차를 실행한다.
- 실패를 무시하지 않는다. 실패가 의도된 상태라면 이유와 다음 조치를 문서화한다.
- 검증 결과는 active plan 또는 완료 문서에 남긴다.

### Release Gate
- 테스트 기준을 충족한다.
- 배포 전략을 명시한다.
- 롤백 경로를 확인한다.
- 운영 모니터링 준비를 완료한다.

## 7. Out Of Scope Without Approval

- 데이터 모델 또는 DB 스키마 대규모 변경
- 인증/인가 정책 변경
- 외부 결제/정산/개인정보 흐름 변경
- 운영비에 큰 영향을 주는 인프라 선택
- 법적/규제 대응 방식 확정
- 프로젝트 구조를 대규모로 재배치하는 작업
- 외부 서비스, SDK, 런타임, 빌드 시스템 신규 도입

## 8. Notes

- 로컬 PC 경로, 개인 툴체인, 특정 IDE 설정은 이 파일에 쓰지 않는다.
- 대신 `docs/15_pc-environment/`에 적고, 실행 계획과 완료 문서에서 해당 파일을 직접 참조한다.
- 외부 참고자료는 `docs/14_references/`에만 둔다.
- 디자인 방향이 아직 확정되지 않았다면 `docs/02_DESIGN.md`는 예시가 아니라 선택지 수준으로 유지한다.
- 구현 단위 작업은 `docs/11_exec-plans/01_active/` 없이 시작하지 않는다.
- 이전 harness 내용이 현재 구조와 맞지 않더라도 바로 삭제하지 않는다. 먼저 현재 구조를 확인하고, 맞지 않는 부분은 “현재 기준”으로 갱신한다.

## 9. Ready

- 문제와 범위가 문서화되어 있다.
- SoT가 지정되어 있다.
- 기능 문장을 1~3개 수준의 상위 문장으로 설명할 수 있다.
- 구현 전 주석만으로도 논리 흐름을 먼저 설명할 수 있다.
- 직접 구현 주석과 호출 중심 주석을 구분해 설계할 수 있다.
- 테스트 또는 검증 방법이 정해져 있다.
- 필요한 로컬 환경이 재현 가능하게 기록되어 있다.

## 10. Done

- 핵심 규칙이 한 곳의 SoT로 수렴했다.
- 구현이 기능 문장과 논리 단위에 맞게 분해되어 있다.
- 세부 동작 설명은 SoT 함수 하나에만 존재한다.
- 상위 함수 주석과 하위 함수 주석이 같은 세부 동작을 중복 설명하지 않는다.
- 주석이 현재 최종 구현과 일치한다.
- 테스트 또는 검증 결과가 남아 있다.
- exec-plan이 정리되어 `docs/11_exec-plans/02_completed/`로 이동했다.
- 남은 기술 부채와 후속 결정이 문서화되어 있다.
