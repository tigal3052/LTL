# Looting The Leviathan - Godot 엔진 기반 구현 기획서 (Prototype-First / TDD)

본 문서는 [기획서v0.2.md](D:/Programming/ex_workspace/LootingTheLeviathan/기획서v0.2.md)의 핵심 의도를 Godot 구현 단계로 옮기기 위한 상세 구현 기획서다.

이 문서에서 MVP는 출시 가능한 최소 제품이 아니라, 전투 감각과 핵심 시스템 가설을 검증하기 위한 `Minimum Viable Prototype`으로 정의한다. 다만 프로토타입은 완전히 버리는 임시물이 아니라, 정식 구현에서 재사용 가능한 `도메인 규칙`, `데이터 포맷`, `테스트 계약`, `텔레메트리 스키마`를 먼저 고정하는 단계로 취급한다.

즉, 프로토타입의 목적은 다음 두 가지를 동시에 달성하는 것이다.

- 전투 감각을 빠르게 검증한다.
- 정식 구현에서 재사용할 수 있는 핵심 로직과 테스트 계약을 먼저 굳힌다.

## 1. 구현 원칙

### 1.1. MVP 정의

- MVP는 출시 가능한 최소 게임이 아니다.
- MVP는 `7:3 전투 화면`, `3x10 지형 그리드`, `파동 읽기`, `에너지 큐`, `장비 배치`, `4핀 시간 압박`, `수리 규칙`이 함께 작동할 때 전투 감각이 성립하는지 검증하는 실험이다.
- MVP의 모든 코드가 폐기 대상인 것은 아니다.
- MVP에서 재사용할 대상은 `순수 도메인 로직`, `데이터 테이블 구조`, `테스트`, `리플레이/로그 규격`이다.
- MVP의 성공 기준은 기능 완성도가 아니라 `정식 구현 사양을 결정할 수 있는 검증 결과와 재사용 가능한 핵심 계약을 확보했는가`이다.

### 1.2. 원 기획의 핵심 구현 축

| 기획 요소 | 프로토타입 구현 관점 | 정식 구현으로 넘길 판단 포인트 |
|---|---|---|
| 거대 생명체 등반/채집 | 적 전투가 아니라 환경을 뚫는 채집 액션 | 전투를 “채집”으로 인식시키는 시각/연출 보강이 필요한가 |
| 파동(Pulse) 타이밍 | 색상과 흐름을 읽고 수동 발사 | 난이도별 파동 속도 구간을 몇 단계로 둘 것인가 |
| 백팩 엔진 빌딩 | 폴리오미노 배치와 시너지로 전투 리듬 변화 | 시너지 종류를 얼마나 늘릴 것인가 |
| 에너지 큐 | 자동 생성 + 수동 발사 리듬 | 큐 용량과 누수 정책을 고정할 것인가 |
| 마력장/체력 이중 내구도 | 속성 매칭과 데미지 효율 차등 | 정식 밸런스에서 계열별 특수 효과를 얼마나 확장할 것인가 |
| 4핀/리미트 타임 | 순수 시간 기반 압박 | 시간 압박 곡선을 스테이지별로 차등화할 것인가 |
| 방해 요소 | 전투 지연과 조준 방해 | 불쾌감이 적은 방해 요소 풀을 어떻게 구성할 것인가 |
| 성장 곡선 | 유물 성장 vs 스테이지 체력 상승 | 난이도와 성장 곡선의 수학적 기준을 어떻게 확정할 것인가 |
| 노드 선택 | 노말 고정 + 랜덤 선택지 | 장기 런에서 이벤트 노드와 우회 전략을 얼마나 강화할 것인가 |
| 보상/재도전 | 최소 보상으로 성장률만 검증 | 정식 보상 루프와 메타 보상을 어떻게 설계할 것인가 |
| 서사/금기 | 프로토타입 단계에서는 보류 | 어느 화면/시점에서 갈등 전환점을 넣을 것인가 |

### 1.3. 기술 설계 원칙

1. **순수 로직 우선:** 파동, 큐, 데미지, 장비 배치, 방해 요소, 노드 생성은 가능한 한 `RefCounted` 기반 순수 로직으로 작성하고 GUT 테스트를 먼저 둔다.
2. **재사용 경계 우선:** 프로토타입 뷰/컨트롤러는 임시로 둘 수 있지만, `domain`과 `data`는 정식 구현에서도 유지 가능한 구조로 작성한다.
3. **데이터 주도 설계:** 데미지 배율, 쿨타임, 핀 소진 시점, 노드 생성 규칙, 성장 곡선은 JSON으로 분리한다.
4. **재현 가능한 결정성:** 고정 seed, 고정 tick, 입력 로그를 기반으로 언제든 같은 결과를 재생할 수 있어야 한다.
5. **정량 로그 우선:** 프로토타입의 판정은 정성 문장보다 수치 로그를 남기는 데 우선순위를 둔다.
6. **TDD 우선:** 정식 구현에 들어가기 전에 프로토타입에서 “무엇이 변하면 안 되는지”를 테스트로 잠근다.

## 3. 프로토타입 MVP 계획

### 3.1. 프로토타입 목표

프로토타입은 아래 네 가지 가설만 검증한다.

1. 파동 읽기와 수동 발사가 재미있는가
2. 장비 배치와 에너지 큐가 전투 리듬을 바꾸는가
3. 시간 압박과 수리 규칙이 납득 가능한 긴장을 만드는가
4. 스테이지 진행에 따른 성장과 난이도 상승이 같이 유지되는가

### 3.2. 프로토타입 입력/전투 규칙 명세

#### 입력과 타게팅

- 마우스 이동 시 3x10 지형 그리드에서 현재 hover 중인 셀 1개를 타겟팅한다.
- 동시에 둘 이상의 셀이 타겟팅되면 안 된다.
- 조준은 마우스 위치에 즉시 반응한다.

#### 발사 규칙

- 클릭 1회당 즉시 1발 발사한다.
- 마우스 버튼을 누르고 있으면 `0.1초` 간격으로 연속 발사한다.
- 발사 시 큐의 맨 앞 에너지를 1개 소비한다.
- 큐가 비어 있을 때 발사 입력이 들어오면 발사는 실패하고 채굴기 내구도가 1 감소한다.

#### 타일 비활성화

- 빔을 맞은 타일은 `0.5초` 동안 비활성화된다.
- 비활성화된 타일을 다시 향해 발사하면 에너지는 소비되지만 판정은 무효 처리된다.
- 따라서 같은 타일에 `0.1초` 간격으로 연속 발사하면 첫 발 이후 최대 4발은 무효화될 수 있다.

#### 수리 규칙

- 빈 큐 상태 발사 등으로 내구도 손실이 3회 누적되면 수리 상태로 진입한다.
- 수리 상태에서는 발사를 할 수 없다.
- 수리 해제 방식은 프로토타입에서는 고정 동작 1종만 사용한다.
- 수리 세부 방식은 `PrototypeTuning.json`에 고정한다.

#### 시간 압박

- 4핀은 오직 시간 경과에 따라 소진된다.
- 방해 요소나 피격은 핀을 직접 제거하지 않는다.
- 기본 프로토타입 기준은 `25% / 50% / 75% / 100%` 경과 시 단계 변화로 둔다.

### 3.3. 파동 이동과 클릭 판정 기준

프로토타입에서는 다음 규칙으로 고정한다.

- 전투 시뮬레이션은 `fixed tick` 기준으로만 계산한다.
- 권장 기본 tick은 `0.05초`다.
- 입력은 발생 시점에 `(tick, target_cell, input_type)` 형태로 기록한다.
- 실제 판정은 해당 입력이 소비되는 고정 tick 시점의 지형 상태를 기준으로 계산한다.
- 같은 `seed + 입력 로그`를 사용하면 결과가 완전히 동일해야 한다.

즉, 프로토타입은 체감보다 재현성과 테스트 가능성을 우선한다.

### 3.4. 프로토타입에 포함할 것과 제외할 것

| 구분 | 포함 | 제외 |
|---|---|---|
| 전투/채집 | 3x10 파동, 에너지 큐, 마력장/체력, 속성 매칭, 조준/발사/수리 | 정식 보스 패턴, 카메라 연출 |
| 인벤토리 | 소형 백팩, 폴리오미노 6~10개, 인접 시너지 | 대규모 아이템 풀, 등급별 세부 밸런스 |
| 성장 | 스테이지 진행에 필요한 최소 보상, 데미지 증가 검증 | 장기 메타 성장, 진화 연출 |
| 시간 압박 | 4핀 소진, 경고, 실패 판정 | 정식 게임오버 연출 |
| 방해 요소 | 빙결 1종, 고장 또는 강풍 1종 | 전체 환경 재해 세트 |
| 노드 진행 | 노말 1개 고정 + 랜덤 선택지 4개 | 장시간 런 구조 |
| 서사 | 없음 | 대사, 컷신, 전환점, 스토리 이벤트 |

### 3.5. 정량 로그 항목

프로토타입은 아래 값을 반드시 기록한다.

- `run_id`
- `seed`
- `stage_index`
- `node_type`
- `shots_fired`
- `shots_hit_match`
- `shots_hit_mismatch`
- `shots_invalid_tile_cooldown`
- `shots_fired_empty_queue`
- `queue_generated`
- `queue_consumed`
- `queue_wasted`
- `durability_loss_count`
- `repair_count`
- `clear_time_sec`
- `fail_time_sec`
- `result`

추가로 계산 가능한 파생 지표:

- 파동 적중률 = `shots_hit_match / shots_fired`
- 미스매치 비율 = `shots_hit_mismatch / shots_fired`
- 무효 발사 비율 = `shots_invalid_tile_cooldown / shots_fired`
- 에너지 누수율 = `queue_wasted / queue_generated`

### 3.6. 프로토타입 단계

#### P0. 테스트 하네스와 로그 기반

- 목표: 프로토타입의 모든 핵심 결과를 seed와 로그로 재현 가능하게 만든다.
- 산출물:
  - `PrototypeTuning.json`
  - 기본 seed 규칙
  - Telemetry 이벤트 목록
  - 입력 로그 포맷
- TDD 대상:
  - 동일 seed 재현
  - 입력 로그 리플레이
  - 로그 필드 누락 방지
- 완료 기준:
  - 동일 seed로 10회 실행 시 노드 생성 결과가 10회 모두 동일하다.
  - 동일 seed + 동일 입력 로그로 10회 재생 시 결과가 10회 모두 동일하다.
  - 로그에 `shots_fired`, `queue_wasted`, `durability_loss_count`, `clear_time_sec`가 기록된다.

#### P1. Headless 코어 루프

- 목표: 화면 없이도 파동, 큐, 데미지, 내구도, 시간 압박이 동작한다.
- 산출물:
  - `PulseModel.gd`
  - `EnergyQueue.gd`
  - `MiningResolver.gd`
  - `RunSimulator.gd`
- TDD 대상:
  - 매치/미스매치 데미지 계산
  - 큐 생성/소비/누수
  - 빈 큐 발사 시 내구도 감소
  - 동일 타일 0.5초 비활성화
  - 4핀 시간 소진
- 완료 기준:
  - 단위 테스트가 모두 통과한다.
  - 빈 큐 상태 발사 3회 후 수리 상태 진입이 재현된다.
  - 동일 타일에 0.1초 간격 연속 발사 시 첫 발 이후 0.5초 동안 판정이 무효 처리된다.
  - `clear`와 `time_over`가 같은 tick에 발생하면 `clear`가 우선된다.

#### P2. 전투 조작 슬라이스

- 목표: 플레이어가 7:3 화면에서 셀을 읽고 직접 조준/발사하는 감각을 검증한다.
- 산출물:
  - `PrototypeMain.tscn`
  - `PrototypeController.gd`
  - `PrototypeHUD.gd`
- TDD 대상:
  - hover 셀 선택
  - 클릭당 1발
  - 홀드 연사 간격
  - 비활성 타일 무효 처리
- 완료 기준:
  - 3x10 전 셀에 대해 hover 시 정확히 1개 셀만 타겟 표시된다.
  - 홀드 발사 평균 간격이 `0.1초 ± 0.02초` 범위에 들어간다.
  - 동일 타일 연속 발사 시 무효 발사 수가 로그에 기록된다.
  - 조준 위치, 발사 tick, 소비된 큐 에너지 색상이 로그에 남는다.

#### P3. 배치/압박/방해 슬라이스

- 목표: 장비 배치가 전투 효율에 영향을 주고, 시간 압박과 방해 요소가 전투를 흔드는지 검증한다.
- 산출물:
  - `InventoryModel.gd`
  - `HazardModel.gd`
  - 방해 요소 1~2종
- TDD 대상:
  - 배치 변경 전후 초당 에너지 생성량 차이
  - 빈 큐 발사로 인한 수리 진입
  - 방해 요소 적용 시 전투 시간 증가
- 완료 기준:
  - 장비 배치 2개 이상 조합에서 초당 에너지 생성량 차이가 로그로 확인된다.
  - 방해 요소 적용 시 평균 클리어 시간 또는 평균 실패 시간이 무방해 조건 대비 증가한다.
  - `repair_count`와 방해 요소 발생 횟수가 로그에 남는다.

#### P4. 미니 런과 성장 곡선 검증

- 목표: 성장과 난이도 상승이 동시에 유지되는지 본다.
- 산출물:
  - `NodeGenerator.gd`
  - 최소 보상 데이터
  - 미니 런 결과 리포트
- TDD 대상:
  - 노드 생성 규칙
  - 성장 곡선 계산 함수
  - 스테이지별 체력/방어막 계산 함수
- 완료 기준:
  - 스테이지별 기대 DPS 증가량과 스테이지별 방어막/체력 증가량이 로그로 계산 가능하다.
  - 스테이지별 평균 클리어 시간과 평균 실패 시간이 모두 기록된다.
  - 같은 조건에서 스테이지가 진행되어도 평균 클리어 시간이 단조롭게 붕괴하지 않는다.
  - 노드 생성 규칙인 `노말 1개 고정 + 랜덤 4개`가 100% 유지된다.

### 3.7. 노드 출현 규칙

프로토타입의 선택 화면은 항상 `5개 선택지`로 구성한다.

- 노말 노드 1개는 무조건 포함한다.
- 나머지 4개는 `취약 / 혼합 / 이벤트` 풀에서 랜덤으로 선택한다.
- 같은 선택 화면 내에서 완전 중복 노드는 허용하지 않는다.
- 이벤트 노드는 최대 1개까지만 허용한다.
- 혼합 노드는 최대 2개까지만 허용한다.
- 남는 슬롯은 취약 노드로 채운다.

권장 기본 가중치:

- 취약: 60%
- 혼합: 30%
- 이벤트: 10%

### 3.8. 성장 곡선 검증 기준

프로토타입은 “유물을 얻고 데미지가 증가하는 비율이 스테이지 방어막/체력 증가를 따라잡는지”를 기록한다.

필수 기록 항목:

- 스테이지별 기대 DPS
- 스테이지별 방어막 총량
- 스테이지별 체력 총량
- 스테이지별 평균 클리어 시간
- 스테이지별 평균 실패 시간
- 스테이지별 승률

이 단계에서는 통과/실패를 자동 판정하지 않는다. 기록된 수치를 보고 사용자가 다음 구현 단계의 밸런싱 방향을 결정한다.

### 3.9. 프로토타입 종료 산출물

- `PrototypeFindings.md`
  - 플레이테스트 로그 요약
  - 핵심 수치 정리
  - 유지/폐기 판단 메모
- `DesignDecisionRecord.md`
  - 유지할 규칙
  - 수정할 규칙
  - 폐기할 규칙
- `TuningBaseline.json`
  - 정식 구현 초기값으로 넘길 수치

## 4. 프로토타입 종료 후 반영 원칙

프로토타입 이후에는 “프로토타입 코드를 확장”하지 않고, “프로토타입에서 검증된 계약을 기준으로 정식 게임을 재구성”한다. 이때 아래 기준으로 분류한다.

### 4.1. 유지할 것

- 테스트로 잠근 순수 도메인 규칙
- JSON 데이터 구조
- 로그 이벤트 이름과 계산식
- 입력/판정/리플레이 규격
- 성장 곡선 계산 함수의 형태

### 4.2. 수정할 것

- 수치값 그 자체
- 큐 용량
- 핀 소진 시점
- 방해 요소 강도
- 보상 드랍 빈도
- 스테이지 체력/방어막 증가 곡선

### 4.3. 교체할 것

- 프로토타입 전용 씬 구조
- 디버그 HUD
- 임시 수리 UI
- 테스트용 연출
- 단일 전투 화면 중심의 임시 흐름

### 4.4. 정식 구현 사양으로 승격할 문서

프로토타입이 끝나면 아래 문서가 정식 구현의 입력값이 된다.

- `PrototypeFindings.md`
- `DesignDecisionRecord.md`
- `TuningBaseline.json`
- 새로 작성할 `ProductionSpec_v1.md`

`ProductionSpec_v1.md`에는 최소한 다음이 포함되어야 한다.

- 프로토타입에서 유지된 규칙 목록
- 수정된 수치와 이유
- 정식 구현에서 추가될 시스템
- 테스트 계층 재편 계획
- 정식 씬/시스템 구조

## 6. TDD 기반 설계 반영 원칙

이 프로젝트에서 TDD는 “테스트 먼저 코드를 짠다” 수준이 아니라, “설계에서 먼저 고정해야 할 계약을 테스트로 잠근다”는 뜻이다.

### 6.1. 테스트 계층

#### 단위 테스트

대상:

- 파동 이동
- 파동 판정
- 데미지 계산
- 큐 생성/소비/누수
- 장비 배치와 시너지
- 방해 요소 상태 변화
- 노드 생성 규칙
- 성장 곡선 계산

목적:

- 수학적 규칙과 상태 전이를 고정한다.

#### 통합 테스트

대상:

- 전투 한 판의 시작부터 종료까지
- 입력 로그 재생
- 수리 진입/해제
- 보상 지급
- 노드 선택 화면 진입

목적:

- 씬과 도메인 계층의 연결이 끊어지지 않도록 한다.

#### 리플레이/회귀 테스트

대상:

- 대표 seed
- 대표 입력 로그
- 대표 성장 곡선 시나리오

목적:

- 밸런스를 만지더라도 규칙 자체가 깨지지 않았는지 확인한다.

### 6.2. 프로토타입 이후 TDD 반영 방식

프로토타입 테스트 이후에는 다음 순서로 정식 구현에 반영한다.

1. 프로토타입에서 검증된 규칙을 `유지 계약`으로 분류한다.
2. 해당 계약을 단위 테스트와 통합 테스트로 다시 고정한다.
3. 프로토타입 뷰를 버리고 정식 뷰를 만들더라도 테스트는 유지한다.
4. 밸런스 수치 조정은 테스트 수정 없이 데이터만 바꿀 수 있게 설계한다.
5. 정식 구현 중 새 기능이 생기면 먼저 실패 테스트를 만들고 최소 구현 후 통과시킨다.

### 6.3. 정식 구현 단계별 TDD 적용 규칙

- **M0:** 프로토타입 결과를 테스트 계약으로 번역한다.
- **M1:** 도메인 모듈별 단위 테스트를 정식 테스트 파일로 고정한다.
- **M2:** 씬 재구성 전 통합 테스트를 먼저 작성한다.
- **M3~M5:** 보상, 성장, 노드, 방해 요소는 데이터 기반 테스트를 먼저 추가한다.
- **M6~M7:** UI와 서사 연결은 스냅샷/흐름 검증 위주로 통합 테스트를 추가한다.
- **M8~M9:** 대표 리플레이를 회귀 테스트로 등록한다.

### 6.4. TDD로 잠글 핵심 계약

정식 구현 전 반드시 테스트로 잠길 항목:

- 파동 tick 규칙
- 발사 소비 규칙
- 빈 큐 내구도 손실 규칙
- 타일 비활성화 규칙
- 수리 진입 조건
- 4핀 시간 소진 규칙
- 노말 1개 고정 + 랜덤 4개 노드 생성 규칙
- 성장 곡선 계산식
- 스테이지 체력/방어막 계산식
- `seed + 입력 로그` 재현 가능성

## 7. 프로토타입 종료 후 재설계 게이트

### 7.1. Gate A: 전투 감각

다음 질문에 답할 수 있어야 한다.

- 파동 읽기와 수동 발사가 조작 재미를 만든다.
- 장비 배치 변경이 전투 효율 차이로 드러난다.
- 시간 압박이 불합리한 벌점이 아니라 전투 긴장으로 느껴진다.
- 수리 규칙이 빈 큐 발사의 비용으로 납득된다.

### 7.2. Gate B: 사양 확정

다음 항목을 문서화한다.

- 파동 tick 속도
- 큐 용량
- 타일 비활성화 시간
- 내구도 손실 횟수와 수리 방식
- 스테이지 체력/방어막 증가식
- 유물 성장식

### 7.3. Gate C: 정식 구현 이월 항목

다음 항목은 프로토타입에서 확정하지 않는다.

- 장기 메타 성장
- 보상 종류 대량 확장
- 서사 전환점
- 대사, 컷신, 스토리 이벤트
- 최종 UI 아트와 연출

## 8. 결론

프로토타입은 정식 게임의 축소판이 아니라, 정식 게임의 핵심 규칙을 먼저 테스트 가능하게 고정하는 단계다.

정식 구현은 프로토타입을 그대로 확장하는 것이 아니라:

- 규칙은 재사용하고
- 수치는 수정하고
- 뷰와 흐름은 재구성하고
